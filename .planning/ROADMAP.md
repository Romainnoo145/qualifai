### Phase 60: Quote Schema Foundation

**Goal**: The Qualifai database and tRPC API fully support quotes, with typed state machines for both Prospect and Quote, and the 3 existing klarifai-core Marfa quotes imported and accessible in Qualifai.

**Depends on**: Nothing (continues from shipped Phase 59)

**Requirements**: DATA-01, DATA-02, DATA-03, DATA-04, DATA-05, DATA-06, DATA-07, DATA-08, DATA-09, DATA-10, FOUND-01, FOUND-02, FOUND-03, FOUND-04, IMPORT-01, IMPORT-02, IMPORT-03, IMPORT-04, TEST-01, TEST-02, TEST-03, TEST-04, TEST-05

**Success Criteria** (what must be TRUE):

1. Romano runs `scripts/import-klarifai-yaml.ts --apply` and sees the 3 Marfa quotes appear in the Qualifai DB with totals matching klarifai-core (OFF001 €7.816,60 / OFF002 €11.495,00 / OFF003 €13.285,80)
2. Calling `quotes.transition(id, 'ACCEPTED')` in a tRPC test sets `Quote.status = ACCEPTED` and `Prospect.status = CONVERTED` in a single DB transaction, with no extra writes required
3. Attempting `admin.updateProspect` with `status: DRAFT` on a prospect currently at `CONVERTED` returns a typed error — the write does not happen
4. A Zod parse of a malformed `snapshotData` object (missing required fields) throws a validation error before the DB write reaches Prisma
5. The multi-project isolation test confirms that an admin token scoped to Project A gets zero results from `quotes.list` when Project B quotes exist

### Phase 61.1: Manual prospect flow polish (INSERTED)

**Goal**: Manual prospect creation produces a viable working prospect: every prospect has a visible logo (Apollo or favicon fallback), the master analyzer survives Gemini API blips via retry+backoff, Romano can retrigger any pipeline stage from the prospect detail page, and AI failures render as friendly messages instead of stack traces.

**Depends on**: Phase 61

**Requirements**: POLISH-01, POLISH-02, POLISH-03, POLISH-04, POLISH-05, POLISH-06, POLISH-07, POLISH-08, POLISH-09, POLISH-10, POLISH-11, POLISH-12, POLISH-13, POLISH-14

**Why this exists** (discovered during Phase 61 smoke testing on Marfa, 2026-04-14):

- Romano added Marfa manually (no Apollo path) and the prospect had no logo. Apollo doesn't even have Marfa, so the Apollo-route would have failed too. Need a non-Apollo favicon fallback for both manual and orphaned prospects.
- The Marfa pipeline rerun hit Gemini 2.5 Pro 503 errors with no retry → narrative analysis silently dropped. The other 5 evidence stages all worked; the only weak link was AI resilience.
- There is no UI to retrigger a pipeline run after a failure. Romano had to ask for a tsx script. This blocks the "fix and retry" loop entirely.
- Prospect cards and detail pages render no logo today — even where `logoUrl` exists from Apollo, there's no `<img>` mounted.

**Success Criteria** (what must be TRUE):

1. Romano adds a fresh prospect via the manual create form (no Apollo) and the prospect card immediately shows a favicon — sourced from Google's `s2/favicons` service if Apollo has no logo, with a HEAD probe + DuckDuckGo fallback + initial-letter avatar as final fallback.
2. Romano triggers a master analysis via the new "Run analyse" button on `/admin/prospects/[id]`. Gemini returns 503 twice in a row. The retry layer waits, retries, and — on third exhaustion — falls back to `gemini-2.5-flash`. The UI shows "AI tijdelijk niet beschikbaar — fallback model gebruikt" instead of a stack trace.
3. Romano clicks "Verrijk opnieuw" / "Run research" / "Run analyse" buttons in the Acties panel and each one shows loading state, completes, and renders a Laatste run timestamp + status. No raw errors leak to the DOM.
4. Existing prospect cards in `/admin/prospects` list now display logos for every prospect — Apollo logoUrl when present, Google favicon URL inline otherwise. No backfill script required.
5. Unit tests cover the master-analyzer retry matrix (3 retries on 503, fallback to flash, immediate bubble on non-retryable), the favicon helper happy/fallback/timeout paths, and the Acties panel error surface (Gemini 503 mock → friendly message rendered).

**Plans**: 4 plans

### Phase 62: Client-Facing Voorstel + PDF Worker

**Goal**: A prospect can open their personalized proposal URL and experience a **modern, web-native proposal** — not a static A4 document in a browser. The web is leading: scroll-driven sections, motion, responsive across devices, and a fresh design language that breaks with klarifai-core's PDF-first thinking. The PDF remains as a print-friendly archival snapshot with the same content, generated by a separate Railway worker.

**Design philosophy** (locked decision Q14, see `klarifai-core/docs/strategy/decisions.md`):

- **Web is primary** — modern SaaS proposal experience inspired by Pitch/Tome/Linear-style scroll narratives, not A4 pages
- **PDF is secondary** — print-optimised flat layout with identical content, web-only interactions excluded
- **Same source of truth** — both render from `Quote.snapshotData`, but with different presentation layers
- **No legacy** — `klarifai-core/docs/design/proposal-template.html` is reference material only, not the design contract

**Depends on**: Phase 61

**Requirements**: DSGN-01, DSGN-02, DSGN-03, CLIENT-01, CLIENT-02, CLIENT-03, CLIENT-04, CLIENT-05, CLIENT-06, CLIENT-07, CLIENT-08, PDF-01, PDF-02, PDF-03, PDF-04, PDF-05, PDF-06, PDF-07

**Success Criteria** (what must be TRUE):

1. **Design system documented before code**: `DESIGN.md` exists with chosen aesthetic, typography, color tokens, motion principles; `UI-SPEC.md` exists as the implementation contract — both produced via the design exploration plans, not lifted from klarifai-core
2. **Web-native experience**: A test prospect opens `/discover/marfa/voorstel` and scrolls through a modern web layout (sticky elements, motion, transitions) — visually unmistakably _not_ a static A4 mockup, and visually unmistakably _not_ the legacy klarifai-core proposal-template.html
3. **Responsive across devices**: The same URL on phone, tablet, and desktop adjusts content priorities per viewport — not just shrinking the desktop layout
4. **Status tracking works invisibly**: First page load for a SENT quote sets `Quote.status = VIEWED` with a timestamp; a second load does not create a second timestamp
5. **Accept flow is transactional**: Prospect clicks "Akkoord", confirms — `Quote.status = ACCEPTED` and `Prospect.status = CONVERTED` are set in a single DB transaction, and Romano receives a Slack notification with a deep link within 30 seconds
6. **PDF is print-friendly, not a screen capture**: After accept, Romano downloads the PDF from the quote detail — it preserves all proposal content (narrative, line items, scope, totals) in a clean print layout, but flattens web-only interactions (animations, motion, hover states) into static representations. It is _not_ a 1:1 visual copy of the live web page.
7. **PDF Worker failure is observable**: When the Railway PDF worker fails after retries, the admin quote detail shows a "PDF mislukt" error indicator rather than a spinner that never resolves

**Suggested Plan structure** (formalised in `/gsd:plan-phase 62`):

- **Plan 62-01: Design discovery** — `/design-consultation` with awesome-design-md as inspiration input → `DESIGN.md` (aesthetic direction, design system foundations, typography, colors, motion principles)
- **Plan 62-02: Visual exploration** — `/design-shotgun` to generate 3-5 design variants in parallel, comparison board, pick committed direction with explicit rationale
- **Plan 62-03: UI-SPEC contract** — `/gsd:ui-phase` to produce `UI-SPEC.md` from the chosen direction (6-pillar design contract: layout, typography, color, components, motion, accessibility)
- **Plan 62-04: Web voorstel implementation** — Build the React/Next.js client-facing proposal page per UI-SPEC (server-rendered, snapshot HTML at SENT, accept flow, status tracking, notifications)
- **Plan 62-05: PDF Worker service** — Separate Railway worker with print-optimised template (NOT the web HTML), idempotent rendering, retry logic, status callbacks. Can run in parallel to 62-04 since the contracts are decoupled.

**Plans**: TBD (formalised by `/gsd:plan-phase 62`, suggested structure above)

**Depends on**: Phase 62

**Requirements**: CONT-01, CONT-02, CONT-03, CONT-04, CONT-05, CONT-06, CONT-07, CONT-08, CONT-09, CONT-10

**Success Criteria** (what must be TRUE):

1. After accepting a quote, Romano opens `/admin/quotes/[id]/contract/new` — the form is prefilled with the prospect's name, quote scope reference, and payment terms placeholder; Romano edits terms, saves, and sees the contract in DRAFT status
2. Romano clicks "Verstuur contract" — the contract transitions to SENT, a snapshot is frozen, and the form becomes read-only
3. The same prospect who accepted the quote opens `/discover/marfa/contract` and sees the contract text; typing their name and clicking "Ondertekenen" records signature, IP address, user-agent, and timestamp in a single transaction

5. Romano clicks "Download PDF" on the signed contract — the downloaded file renders the full contract text with the signature block, generated via the same Railway worker as proposal PDFs
