1. Create `lib/analysis/types.ts` with the MasterAnalysis output type contract:
   - `AnalysisKPI`: { label: string, value: string, context: string } — prospect-relevant scale metrics
   - `AnalysisContext`: { hook: string, kpis: AnalysisKPI[] (exactly 3), executiveHook: string }
   - `AnalysisTrigger`: { category: 'market' | 'compliance_esg' | 'capital_derisking', title: string, narrative: string, numbers: string[], urgency: 'high' | 'medium' | 'low' }
   - `AnalysisTrack`: { spvName: string, spvCode: string, scope: string, relevance: string, strategicTags: string[] }
   - `MasterAnalysis`: { version: 'analysis-v1', context: AnalysisContext, triggers: AnalysisTrigger[] (exactly 3), tracks: AnalysisTrack[] (2-3), generatedAt: string, modelUsed: string }
   - Export `MasterAnalysisInput` type: { intentVars: IntentVariables, passages: RagRetrievedPassage[], prospect: AnalysisProspectProfile, spvs: AnalysisSPV[] }
   - `AnalysisProspectProfile`: { companyName: string, industry: string | null, description: string | null, specialties: string[], country: string | null, city: string | null, employeeRange: string | null, revenueRange: string | null }
   - `AnalysisSPV`: { name: string, code: string, slug: string, metricsTemplate: unknown }

2. Create `lib/analysis/master-prompt.ts` with `buildMasterPrompt(input: MasterAnalysisInput): string`:
   - System preamble: "Je bent een senior strategisch adviseur die boardroom-ready partnership analyses schrijft voor Europe's Gate. Toon: zakelijk-visionair Nederlands — clean business language, geen anglicismen, confident maar niet stijf. Schrijf alsof het een McKinsey board-presentatie is."
   - CRITICAL tone rules in prompt:
     - Use "uw" for professional distance, prospect name in hooks/headers
     - Numbers must be prospect-relevant (steel capacity they'd use, waste reduction in their sector) — NOT project-grandeur marketing stats
     - When RAG lacks hard numbers: write qualitative trigger with directional evidence, never fabricate
     - When prospect data conflicts with RAG data: show both perspectives
     - NEVER mention AI, RAG, scraping, embeddings, machine learning, or any technical pipeline terminology
   - Section 1 (Context): Build from intent variables (sector_fit, operational_pains primarily), ask for prospect-specific hook (2-3 sentences: setup + insight + relevance), 3 KPI blocks that connect RAG numbers to prospect operations, executive hook tying their pain to Atlantis
   - Section 2 (Triggers): Build from intent categories mapped to trigger types:
     - market trigger: from sector_fit + investment_growth signals
     - compliance_esg trigger: from esg_csrd signals
     - capital_derisking trigger: from investment_growth + operational_pains
     - For each: include specific numbers from RAG passages, frame in prospect's context
   - Section 3 (Partnership): Include list of available SPVs with their metricsTemplate data, ask AI to select top 2-3 most relevant based on intent vars + RAG passages, generate scope + relevance per track. No CTA — contact toggle already exists.
   - Include RAG passages as numbered references: "[1] {content} (bron: {documentTitle})"
   - Include intent variable signals as structured input per category
   - Output format: strict JSON matching MasterAnalysis type (instruct AI to return valid JSON only)

3. Add `ProspectAnalysis` model to `prisma/schema.prisma`:

   ```
   model ProspectAnalysis {
     id            String   @id @default(cuid())
     createdAt     DateTime @default(now())
     updatedAt     DateTime @updatedAt
     version       String   @default("analysis-v1")
     content       Json     // MasterAnalysis object

     researchRunId String      @unique

     prospectId    String
     prospect      Prospect    @relation(fields: [prospectId], references: [id], onDelete: Cascade)

     @@index([prospectId])
     @@index([researchRunId])
   }
   ```

   - Add `prospectAnalyses ProspectAnalysis[]` relation to Prospect model
   - Add `prospectAnalysis ProspectAnalysis?` relation to ResearchRun model
