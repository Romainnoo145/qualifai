# Phase 61.4: Admin Design System Sweep - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Apply DESIGN.md cool SaaS design tokens across all 15 admin pages. Replace hardcoded Tailwind colors, ad-hoc spacing, and inconsistent component patterns with the documented token vocabulary. Redesign the sidebar to match the DESIGN.md §3.2 wireframe. Extract repeated UI patterns into shared components.

**Not in scope:** Layout restructuring of individual pages (only token application to current layouts). New features or functionality. Client `/voorstel` surface (separate phase 62).

</domain>

<decisions>
## Implementation Decisions

### Token migration strategy

- Sweep all 15 admin pages in one phase — no half-migrated state
- First plan: sync `globals.css` to DESIGN.md §3.2 — audit and add missing CSS variables (sidebar colors, hover states, muted text, canvas/surface values)
- Replacement approach: CSS variables via Tailwind arbitrary values (e.g., `bg-slate-50` → `bg-[var(--color-surface)]`, `text-gray-500` → `text-[var(--color-muted)]`)
- Full typography alignment: audit font sizes, weights, letter-spacing across all admin pages to match DESIGN.md scale (14-15px Sora 300 body, 10-11px Sora 500 uppercase eyebrows, etc.)

### Admin visual identity — CTA & gold usage

- **Primary CTA buttons:** deep navy (#0a0a2e) solid fill, white text, pill shape — NOT the gold gradient (gold gradient is reserved for client `/voorstel` surface only)
- **Secondary buttons:** 1px navy border, white/transparent bg, navy text
- **Gold in admin:** micro-accent ONLY — sidebar active dot, heading period, rare badge highlight. Never on button backgrounds, section backgrounds, or borders
- **Clear surface split:** gold = client-facing brand warmth, navy = admin professionalism

### Sidebar redesign

- Rebuild sidebar to match DESIGN.md §3.2 wireframe: 58px wide, icon-only nav, navy logo square with gold K
- Expand-on-hover for text labels
- Active item: gold indicator dot/bar
- White background with subtle zinc right border

### Component extraction

- Extract patterns that appear 3+ times with near-identical markup: page header (title + breadcrumb + actions), stat card, data table wrapper, empty state — approximately 4-6 new components
- Components live in `components/ui/` alongside existing `glass-card.tsx`, `button.tsx`
- Keep existing page layouts — apply tokens to current structure, don't restructure

### Status & state styling

- Color-coded subtle backgrounds per state category:
  - Green tint: positive states (Bekeken, Geaccepteerd, Voltooid)
  - Amber tint: pending states (Concept, Verstuurd, Wacht)
  - Red tint: negative states (Mislukt, Afgewezen)
- Unify across all badge/pill/chip instances: prospect status, quote status, pipeline stage, quality indicators
- Consolidate into a single `status-badge.tsx` component (or update existing)

### Claude's Discretion

- Exact grouping of the 279 hardcoded color classes into token mappings
- Which ~4-6 components to extract (based on actual repetition count)
- Plan ordering and page grouping within the sweep
- Handling edge cases where current layout is incompatible with token application

</decisions>

<specifics>
## Specific Ideas

- Gold gradient CTAs are the Klarifai brand signature — they stay on `/voorstel` client pages but are explicitly NOT used in admin. Admin gets navy buttons.
- Gold in admin is a whisper, not a shout: sidebar active dot, heading period, maybe one badge highlight per screen. That's it.
- The sidebar redesign to 58px icon-only nav is the single biggest visual change — it sets the admin's identity apart from the client surface.
- User prefers compact UI (p-6 not p-10), data-dense — the token sweep should not increase spacing beyond what DESIGN.md prescribes.

</specifics>
