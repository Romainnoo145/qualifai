## Summary

The admin codebase is in a mixed state: `globals.css` was already revised on 2026-04-17 to adopt the Cool SaaS direction (zinc surfaces, navy ink, gold gradient CTA) with a well-structured `@theme` block and a solid set of `.admin-btn-*`, `.admin-toggle-*`, `.admin-state-*`, and `.input-minimal` utility classes. However, the individual page files were not updated in lock-step. They still carry hundreds of hardcoded Tailwind color classes from the pre-design-system era: `bg-slate-*`, `text-[#040026]` literals, `bg-[#EBCB4B]` hex values, `bg-emerald-*`, `bg-blue-*`, `bg-purple-*`, and non-token typography like `font-black`, `tracking-tighter`, `rounded-xl`, `rounded-2xl`.

The inspection reveals 23 page files across 15+ admin routes (the CONTEXT lists 15 targets but actual count is 20+ including briefs, hypotheses, research, campaigns/new). The heaviest offenders by hardcoded-class count are: `outreach/page.tsx` (69), `campaigns/new/page.tsx` (61), `contacts/[id]/page.tsx` (46), `prospects/page.tsx` (44), `campaigns/[id]/page.tsx` (37), `settings/page.tsx` (35), `page.tsx` dashboard (30). The `prospects/[id]/page.tsx` detail page is the most recent and cleanest — it already uses CSS variable arbitrary values throughout.

The sidebar in `layout.tsx` already matches the 58px rail spec and uses CSS variables correctly — it is largely done and needs only minor cleanup (the `expand-on-hover` behavior is missing). The `globals.css` token vocabulary covers most needs but is missing one mapping: the CONTEXT.md locked decision states primary CTAs should be **navy solid fill** (not gold gradient), yet `globals.css` has `admin-btn-primary` as the gold gradient and `admin-btn-secondary` as white. This inversion must be reconciled as the first globals.css delta.

**Primary recommendation:** Run the sweep in four waves — (1) globals.css delta + token mapping table, (2) extract 5 shared components, (3) page-by-page token migration grouped by similarity, (4) status badge unification.

### Tailwind v4 Key Behavior (HIGH confidence — observed in codebase)

The project uses Tailwind CSS v4 with the `@theme { }` block in `globals.css`, NOT `tailwind.config.ts`. This means:

- CSS variables defined in `@theme {}` are automatically available as Tailwind utilities: `--color-surface` → `bg-surface`, `text-surface`
- Arbitrary values `bg-[var(--color-surface)]` also work and are used in existing files
- `@layer components {}` defines reusable CSS classes (`.glass-card`, `.admin-btn-primary`, etc.)
- No `tailwind.config.ts` custom tokens are needed — everything lives in `globals.css`

**The recommended pattern for this phase** is Tailwind arbitrary values: `bg-[var(--color-surface)]` rather than native Tailwind token names, because the existing pages use this pattern consistently and it makes token origin explicit.

### Pattern 1: Token Mapping Table

**What:** Systematic mapping of hardcoded classes to CSS variable equivalents.
**When to use:** Before touching any page file, define the complete substitution table in globals.css comments or a separate mapping reference.

```typescript
// Source: codebase inspection + DESIGN.md §3.2 + globals.css @theme

// SURFACES
'bg-slate-50'         → 'bg-[var(--color-surface-2)]'   // zinc-100
'bg-[#FCFCFD]'        → 'bg-[var(--color-surface)]'     // white
'bg-white'            → 'bg-[var(--color-surface)]'
'bg-slate-50/50'      → 'bg-[var(--color-surface-2)]'

// TEXT
'text-[#040026]'      → 'text-[var(--color-ink)]'       // note: #040026 = ink-deep variant
'text-[#0a0a2e]'      → 'text-[var(--color-ink)]'
'text-slate-400'      → 'text-[var(--color-muted)]'
'text-slate-500'      → 'text-[var(--color-muted)]'
'text-slate-300'      → 'text-[var(--color-muted)]'     // or --color-border-strong for borders

// BORDERS
'border-slate-100'    → 'border-[var(--color-border)]'
'border-slate-200'    → 'border-[var(--color-border-strong)]'
'border-[#EBCB4B]'    → 'border-[var(--color-gold)]'    // gold borders on CTAs

// BUTTONS (inline patterns → CSS classes)
'bg-[#EBCB4B] text-[#040026] border-[#EBCB4B]'   → 'admin-btn-primary'
'bg-slate-50 text-slate-400 border-slate-100'     → 'admin-btn-secondary disabled:...'

// SEMANTIC (activity tags already mapped in globals.css — use tag classes)
'bg-emerald-50 text-emerald-500'  → 'admin-state-success' (via .admin-state-pill)
'bg-blue-50 text-blue-500'        → 'admin-state-info'
'bg-amber-* text-amber-*'         → 'admin-state-warning'
'bg-red-50 text-red-500'          → 'admin-state-danger'
'bg-purple-50 text-purple-500'    → 'admin-state-accent'

// TYPOGRAPHY (legacy → token-aligned)
'font-black'                        → 'font-bold'  (700 — DESIGN.md: 700 = headings only)
'tracking-tighter'                  → 'tracking-[-0.025em]' or per-context
'tracking-widest'                   → 'tracking-[0.14em]' or use 'admin-eyebrow' class
'text-[10px] font-black uppercase'  → 'admin-eyebrow'
'text-[9px] font-bold uppercase'    → 'admin-eyebrow' (11px min per DESIGN.md)

// RADIUS (legacy non-token radii)
'rounded-xl'   → 'rounded-[var(--radius-md)]'   // 12px
'rounded-2xl'  → 'rounded-[var(--radius-lg)]'   // 16px
'rounded-3xl'  → 'rounded-[var(--radius-xl)]'   // 20px
'rounded-md'   → 'rounded-[var(--radius-xs)]'   // 6px
```

### Anti-Patterns to Avoid

- **`font-black` (weight 900):** DESIGN.md specifies 700/500/300 only. `font-black` = weight 900 which is not loaded from Google Fonts (only 300/500/700 loaded). Replace with `font-bold` (700) for headings, `font-medium` (500) for labels.
- **`rounded-xl`, `rounded-2xl` in-page:** Use `rounded-[var(--radius-md)]` etc. for token traceability.
- **`tracking-widest` in-page:** 0.1em, not identical to DESIGN.md eyebrow 0.14–0.18em. Use `.admin-eyebrow` class or explicit `tracking-[0.14em]`.
- **`bg-[#FCFCFD]`:** This specific hex appears 3+ times as a pseudo-white. Map to `bg-[var(--color-surface)]`.
- **`text-[#007AFF]` hover states:** Apple Blue is defined in DESIGN.md §2.1 as "focus ring only". Using it as hover link color violates the spec. Replace link hovers with `hover:text-[var(--color-ink)]` or `hover:opacity-70`.
- **`ui-tap`, `ui-focus` classes:** These appear in several files but are NOT defined in `globals.css` (confirmed by grep). They are undefined CSS classes producing no effect. Remove them during sweep — do not preserve.
- **`shadow-inner`, `shadow-xl`:** DESIGN.md §2.6 says "almost never" shadows. Remove all decorative shadows. Only `.glass-card-hover:hover` box-shadow is permitted.
- **`animate-fade-in-up` on page wrappers:** Fine to keep — defined in globals.css and serves the 200ms ease-out micro-motion spec.

## Common Pitfalls

### Pitfall 1: `#040026` vs `#0a0a2e` — two navy shades used inconsistently

**What goes wrong:** The codebase uses both `text-[#040026]` (`--color-ink-deep`) and `text-[#0a0a2e]` (`--color-ink`) interchangeably for body text. DESIGN.md defines `--color-ink: #0a0a2e` as the primary text color and `--color-ink-deep: #040026` as "brochure container top — available for dark surfaces only."
**Why it happens:** Some pages were written before the distinction was codified.
**How to avoid:** Map ALL `text-[#040026]` → `text-[var(--color-ink)]` for body/heading text. Reserve `--color-ink-deep` for dark surface overlays only. The distinction is imperceptible in practice but keeps tokens semantically correct.
**Warning signs:** Grep for `\[#040026\]` and `\[#0a0a2e\]` in the same file.

### Pitfall 2: `btn-pill-primary` vs `admin-btn-primary` — two button class families

**What goes wrong:** `globals.css` defines two separate button families — `btn-pill-*` (brochure surface, gold gradient, pill shape) and `admin-btn-*` (admin surface, currently gold gradient but should become navy). Some pages use `btn-pill-yellow` for admin CTAs, which is correct gold behavior but wrong after the CTA color switch.
**Why it happens:** Both families exist and look similar before the navy CTA correction.
**How to avoid:** During the globals.css delta, update only `admin-btn-primary`. Leave `btn-pill-*` classes untouched (brochure uses them). Audit all admin pages for `btn-pill-yellow` usage and replace with `admin-btn-primary`.
**Warning signs:** `btn-pill-yellow` and `btn-pill-primary` in admin page files.

### Pitfall 3: `ui-tap` / `ui-focus` are undefined ghost classes

**What goes wrong:** Several pages apply `ui-tap ui-focus` to interactive elements, but these classes are not defined in `globals.css` (confirmed by grep). They silently do nothing.
**Why it happens:** Legacy from an older design system iteration that was never fully implemented.
**How to avoid:** Remove these class names during sweep. Do not try to define them — they were placeholder interaction utilities that were superseded by the `.admin-btn-*` transition system.
**Warning signs:** Any className string containing `ui-tap` or `ui-focus`.

### Pitfall 4: `font-black` loads weight 900 which is NOT in the Google Font load

**What goes wrong:** `app/layout.tsx` loads Sora weights 300, 500, 700 only. `font-black` (weight 900) causes the browser to synthesize a bold approximation — looks slightly off and violates DESIGN.md.
**Why it happens:** Fast prototyping used `font-black` as "maximum bold".
**How to avoid:** Replace all `font-black` with `font-bold` (maps to 700, which IS loaded). For page titles that need maximum impact, use 700 + tight tracking.
**Warning signs:** Any `font-black` in admin page TSX.

### Pitfall 5: Status badge rewrite must also cover QuoteStatus (not just ProspectStatus)

**What goes wrong:** The current `status-badge.tsx` covers prospect states. Phase 61 added `QuoteStatus` (DRAFT/SENT/VIEWED/ACCEPTED/REJECTED/EXPIRED/ARCHIVED). There is a separate `QuoteStatusBadge` component in `components/features/quotes/quote-status-badge.tsx`. Unification means merging both into the single `status-badge.tsx`.
**Why it happens:** Quote badge was built separately in Phase 61 without yet having this sweep phase.
**How to avoid:** During status badge rewrite, add all QuoteStatus values to the STATUS_CLASS mapping and delete the separate quote-status-badge file (or re-export from status-badge.tsx for backward compat).
**Warning signs:** Import of `QuoteStatusBadge` from `components/features/quotes/` — should become `StatusBadge` from `components/ui/`.

### Pitfall 6: `npm run check` will flag unused CSS classes if globals.css is updated before page sweep

**What goes wrong:** If `.admin-btn-primary` changes to navy-solid but pages still reference old inline gold-gradient patterns, both exist simultaneously during the sweep. This is fine — but be aware that `npm run check` may flag type errors unrelated to design (not CSS-related for Next.js/TypeScript).
**How to avoid:** Run `npm run check` after each plan, not just at phase end. Fix TypeScript errors immediately; CSS drift is tolerable mid-sweep.
