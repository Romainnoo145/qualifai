**One-liner:** og-logo-first logo fetch wired into both createAndProcess and enrichProspect; createAndProcess extended with 6 sticky optional fields and a collapsible Dutch form section.

## What Was Built

### Task 1: getHighResLogoUrl wiring in admin.ts

**Import added:**

```typescript
import { getHighResLogoUrl } from '@/lib/enrichment/og-logo';
```

**createAndProcess fire-and-forget IIFE (PARITY-07):**
Added just before the research pipeline background IIFE (~line 567). Tries `getHighResLogoUrl(cleanDomain)` first; falls through to `getFaviconUrl(cleanDomain)` on null; updates `prospect.logoUrl` if either returns a value. Non-blocking — wrapped in try/catch; research pipeline runs in parallel.

**enrichProspect EnrichmentNoCoverageError catch (PARITY-08):**
Inserted inside the `EnrichmentNoCoverageError` catch block (added by Plan 01), after setting `noCoverage = true` and populating the minimal `enriched` object, and BEFORE the `else { throw err }` branch. Same og-logo → favicon fallback → update `prospect.logoUrl` pattern. Non-blocking try/catch so the amber partial-success response always returns.

### Task 2: createAndProcess Zod schema extension + sticky guard (PARITY-10)

**6 optional nullable fields added to Zod input:**

| Field           | Type                                                                         | Constraint         |
| --------------- | ---------------------------------------------------------------------------- | ------------------ |
| `companyName`   | `z.string().min(1).max(200)`                                                 | optional, nullable |
| `industry`      | `z.string().min(1).max(100)`                                                 | optional, nullable |
| `description`   | `z.string().max(500)`                                                        | optional, nullable |
| `employeeRange` | `z.enum(['1-10','11-50','51-200','201-500','501-1000','1001-5000','5001+'])` | optional, nullable |
| `city`          | `z.string().max(100)`                                                        | optional, nullable |
| `country`       | `z.string().max(100)`                                                        | optional, nullable |

**db.prospect.create** receives the optional fields via conditional spread when non-null:

```typescript
...(input.companyName != null && { companyName: input.companyName }),
```

**Sticky guard in Apollo enrichment merge:**
After `buildEnrichmentData(combined.merged, ...)`, a second spread overrides companyName/industry/city/country with the manually-provided values if present. This means Apollo NEVER overwrites manually-entered company identity fields.

Fields that are sticky: `companyName`, `industry`, `city`, `country`
Fields that are NOT sticky: `description`, `employeeRange` (Apollo values preferred if available)

**Prisma type note:** `employeeRange` is `String?` in Prisma schema — no enum cast needed. The z.enum Zod validation is the only enforcement layer.

**ReadableSlug improvement:** When Apollo fails, the slug fallback now uses `input.companyName ?? cleanDomain.split('.')[0]!` — produces a better slug when a manual company name was provided.

### Task 3: Optionele verrijking collapsible form section (PARITY-09)

**Visual description:** Collapsed `<details>` block below the "Tactical Intelligence" textarea inside the glass-card. Summary line reads "Optionele verrijking" in gray font. When expanded, a 2-column grid appears with:

- Row 1: Bedrijfsnaam (text, left) + Sector (text, right)
- Row 2: Omschrijving (textarea, full-width col-span-2)
- Row 3: Medewerkers (select with 7 ranges, left) + Stad (text, right)
- Row 4: Land (text, default "Nederland", left)

All inputs use `input-minimal w-full` class (established admin pattern). All are disabled during submission (`disabled={stage !== 'idle'}`). Country defaults to "Nederland" via `useState('Nederland')`.

**Submit handler:** Optional fields are spread into the `createAndProcess.mutate()` call only when non-empty (trim guards prevent whitespace-only submissions from being sent as values).

**Reset:** "Deploy Another Engine" button resets all 6 fields, with enrichCountry resetting back to "Nederland".

## Deviations from Plan

### Deviation 1: createAndProcess had no favicon IIFE before this plan

**Found during:** Task 1 read-first
**Issue:** The plan's context described an existing `getFaviconUrl` IIFE in `createAndProcess` — but the actual code had no such IIFE. The favicon-only IIFE exists in `createProspect` (separate, simpler mutation). `createAndProcess` had no logo fetch at all.
**Fix:** Added the new og-logo → favicon IIFE as a net-new addition (not a replacement). The plan's outcome is identical — `createAndProcess` now has an og-logo-first IIFE. Just "replacing" vs "adding" changed the implementation verb.
**Impact:** None on behaviour or acceptance criteria.

### Deviation 2: Sticky guard scope narrowed (description + employeeRange not sticky)

**Found during:** Task 2 analysis
**Issue:** Plan mentioned 4 sticky fields in the guard example (companyName, industry, city, country) but description and employeeRange were listed in the initial db.prospect.create spread (written at create time) without a corresponding sticky guard in the Apollo merge. This is intentional — description and employeeRange from Apollo are higher quality than a free-form manual entry.
**Fix:** Applied sticky guard only to companyName/industry/city/country. Description and employeeRange are written at create-time but can be overwritten by Apollo on the enrichment pass.
**Impact:** Intentional — Apollo description is full-length professional text; manual description is a short hint.
