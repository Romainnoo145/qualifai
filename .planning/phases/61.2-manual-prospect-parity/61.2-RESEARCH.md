## Summary

Phase 61.2 is a code-audit-and-fix phase, not a greenfield build. All four decisions (D1–D4) work against an already-known, already-traced codebase. Every critical callsite has been located. The risk surface is narrow: one Apollo catch site, one new file (og-logo.ts), one form extension, and one component (ProspectEnrichmentBadge).

**Apollo error path:** The 422 throw originates at `lib/enrichment/providers/apollo.ts:470` inside `apolloFetch`, propagates through `enrichCompanyWithFallbackQueries` (re-thrown at line 521), re-thrown again by `apolloProvider.enrichCompany` at line 540, caught by `runWithWaterfall` in `service.ts` at line 190, and re-thrown as `[enrichment:enrichCompany] all providers failed. apollo: Apollo API error (422): ...` — reaching `admin.enrichProspect` as an unhandled error that currently becomes a red toast. The fix catches at the `apolloProvider.enrichCompany` level (returning sentinel) and at `enrichProspect` mutation (catching and returning partial-success).

**og-logo.ts:** jsdom 27.4.0 is already in `devDependencies` and `vitest.config.ts` sets environment to `jsdom`. For server-side parsing in og-logo.ts, prefer a minimal regex approach (no new deps) since og/twitter meta tags are predictable single-line HTML attributes. Plain `fetch` with 5s AbortSignal.timeout is preferred over the scrapling service for homepage fetches — scrapling's 30s timeout is excessive for a meta-tag scrape, and small Dutch SMB homepages don't require stealth bypassing.

**Form:** The existing `/admin/prospects/new` page uses `api.admin.createAndProcess` (the full Apollo+research pipeline), NOT `api.admin.createProspect`. The "Optionele verrijking" section must add optional fields to `createAndProcess` (not createProspect), since that is what the page calls. The sticky rule applies: if `companyName` is provided, the Apollo enrichment catch block must not overwrite it.

**Primary recommendation:** Four plans, three waves. Wave 1: Apollo sentinel + og-logo. Wave 2: Manual enrichment form. Wave 3: Render audit + badge + human-verify.

## Architecture Patterns

### Apollo Error Sentinel Pattern

**What:** Return a typed sentinel value instead of throwing when Apollo returns 422/no-coverage. The sentinel propagates upward as partial-success, not error.

**When to use:** Any time Apollo returns HTTP 422 with "Required parameter" or "no org found" message body.

**Implementation site:** `lib/enrichment/providers/apollo.ts::enrichCompanyWithFallbackQueries`

The current flow:

```
apolloFetch → throws Error("Apollo API error (422): ...")
  ↓ caught in enrichCompanyWithFallbackQueries loop (line 520-523) as lastError
  ↓ sawSuccessfulResponse = false, lastError is set
  ↓ line 525-527: if (!sawSuccessfulResponse && lastError) throw lastError
  ↓ propagates through apolloProvider.enrichCompany (line 540: await enrichCompanyWithFallbackQueries)
  ↓ runWithWaterfall catch (service.ts:190-200): errors.push; errorCount++
  ↓ line 216-218: throw new Error("[enrichment:enrichCompany] all providers failed...")
  ↓ admin.enrichProspect line 251: unhandled → red error toast
```

The fix at `enrichCompanyWithFallbackQueries`:

```typescript
// Check if all 422 errors are "no coverage" (not API key issues)
const isNoCoverage = /422|no org found|Required parameter/i.test(
  lastError.message,
);
if (isNoCoverage) {
  return null; // sentinel: caller returns minimal EnrichedCompanyData
}
throw lastError;
```

Then in `apolloProvider.enrichCompany`, if `enrichCompanyWithFallbackQueries` returns null, return a minimal sentinel object:

```typescript
if (!organization) {
  // Check if this was a no-coverage case (null returned vs throw)
  return {
    domain: clean,
    companyName: null,
    industry: null,
    subIndustry: null,
    // ... all other fields null, _noCoverage: true marker
  } as EnrichedCompanyData & { _noCoverage: true };
}
```

Alternative approach (cleaner): Introduce `EnrichmentNoCoverageError` class (similar to `EnrichmentPlanLimitedError`) and catch it specifically in `admin.enrichProspect`.

**Recommended approach:** `EnrichmentNoCoverageError` class — mirrors the existing `EnrichmentPlanLimitedError` pattern exactly. The catch lives in `admin.enrichProspect`.

### og-logo.ts Pattern

**What:** A pure async function that GETs a homepage, regex-parses the HTML for 4 meta-tag patterns in priority order, HEAD-probes the best candidate URL, and returns the first confirmed URL.

```typescript
// lib/enrichment/og-logo.ts
const TIMEOUT_MS = 5000;

export async function getHighResLogoUrl(
  domain: string,
): Promise<string | null> {
  const normalized = normalizeDomain(domain);
  if (!normalized) return null;

  const htmlResult = await fetchHomepageHtml(normalized);
  if (!htmlResult) return null;

  const candidates = extractLogoUrlCandidates(htmlResult, normalized);
  for (const candidateUrl of candidates) {
    if (await headProbeWithContentLength(candidateUrl)) return candidateUrl;
  }
  return null;
}
```

**Regex patterns** (no new deps):

```
og:image:     /<meta[^>]+property=["']og:image["'][^>]+content=["']([^"']+)["']/i
twitter:image: /<meta[^>]+name=["']twitter:image["'][^>]+content=["']([^"']+)["']/i
apple-touch-icon: /<link[^>]+rel=["']apple-touch-icon["'][^>]+href=["']([^"']+)["']/ig  (pick largest sizes= attr)
icon png:     /<link[^>]+rel=["'](?:shortcut )?icon["'][^>]+type=["']image\/png["'][^>]+href=["']([^"']+)["']/ig
```

Note: `og:image` and `twitter:image` can appear in either attribute order (`property=... content=...` or `content=... property=...`). The regex should handle both orderings.

**HEAD probe with content-length:**

```typescript
async function headProbeWithContentLength(url: string): Promise<boolean> {
  try {
    const res = await fetch(url, {
      method: 'HEAD',
      signal: AbortSignal.timeout(TIMEOUT_MS),
    });
    const cl = Number(res.headers.get('content-length') ?? '0');
    return res.ok && cl > 0;
  } catch {
    return false;
  }
}
```

## Common Pitfalls

### Pitfall 1: og:image attribute order

**What goes wrong:** Some sites write `<meta content="..." property="og:image">` (content before property). A naive regex anchored to `property=` then `content=` will miss these.
**How to avoid:** Use two capture attempts per tag, or use a more flexible regex that finds both attributes regardless of order. Parse the whole tag string and extract both attributes separately.

### Pitfall 2: Apollo 422 is retried 3 times before failing

**What goes wrong:** `apolloFetch` already retries 3 times with backoff on all errors except 429. A 422 no-coverage response is retried twice unnecessarily, adding ~2s delay.
**How to avoid:** Inside `apolloFetch`, check if `response.status === 422` and break the retry loop early (same pattern as the 429 case). Or just accept the 2s overhead since this is a rare case.

### Pitfall 3: createAndProcess vs createProspect

**What goes wrong:** The `/admin/prospects/new` page calls `api.admin.createAndProcess`, NOT `api.admin.createProspect`. Adding optional fields only to `createProspect` will have no effect on the create form.
**How to avoid:** Extend `createAndProcess` Zod input with the optional enrichment fields AND add the sticky logic (skip Apollo overwrite if companyName provided) inside createAndProcess's Apollo try/catch block.

### Pitfall 4: Relative URL resolution for og:image

**What goes wrong:** Some sites return `/og-image.png` or `//cdn.example.com/logo.png` as og:image content. A bare regex match without URL resolution will produce a broken URL.
**How to avoid:** After extracting the href/content value, resolve it against `https://${normalized}` using `new URL(value, `https://${normalized}`)`.

### Pitfall 5: HEAD probe content-length is not always set

**What goes wrong:** Some CDNs omit `content-length` on HEAD responses even for valid images. The probe would incorrectly return false.
**How to avoid:** Accept `res.ok` even if content-length is 0 or missing — only reject on non-2xx status or network error. The requirement says "non-zero content-length" but in practice, treat `res.ok && cl >= 0` as passing (omitted = assume ok).

### Pitfall 6: createAndProcess form page state

**What goes wrong:** The new/page.tsx uses `createAndProcess.mutate({ domain, internalNotes })`. When adding optional fields, the form must reset them on success. The existing reset block at line 162 only clears `domain` and `notes` — it must also clear the new optional field states.
**How to avoid:** Add the new useState variables to the reset block explicitly.

## Key Investigation Answers

### Q1: Where exactly does the Apollo error bubble up?

**Call chain (all confirmed by codebase read):**

1. `apolloFetch` (`apollo.ts:468-471`) — throws `Error("Apollo API error (422): {\"error\":\"Required parameter 'domain' missing\"}")` when `!response.ok`
2. `enrichCompanyWithFallbackQueries` (`apollo.ts:501-523`) — catches into `lastError`, loop exhausts 3 queries, `sawSuccessfulResponse = false` → line 525: `throw lastError`
3. `apolloProvider.enrichCompany` (`apollo.ts:535-548`) — `await enrichCompanyWithFallbackQueries(...)` at line 540 re-throws
4. `runWithWaterfall` callback in `service.ts` (`service.ts:267-277`) — the callback `provider.enrichCompany(domain, prospectId)` throws; caught at line 190; `errors.push(\`apollo: ${message}\`)`; `errorCount++`
5. `service.ts:216-218` — `throw new Error("[enrichment:enrichCompany] all providers failed. apollo: Apollo API error (422): ...")`
6. `admin.enrichProspect` (`admin.ts:251`) — `await enrichCompany(...)` re-throws; no catch → tRPC sends error response → red toast in Acties panel

**The EXACT catch site for D1:** Two options:

- Option A: `enrichCompanyWithFallbackQueries` (line 525): detect 422 no-coverage before re-throwing → return null → `apolloProvider.enrichCompany` receives null organization → return minimal EnrichedCompanyData
- Option B: Introduce `EnrichmentNoCoverageError` class in `service.ts` (mirrors EnrichmentPlanLimitedError), add detection marker in runWithWaterfall, catch in `admin.enrichProspect`

**Recommendation: Option B** — mirrors exact existing pattern. Less surgery on apollo.ts internals.

### Q2: Should og-logo use scrapling or plain fetch?

**Decision: Plain fetch with `AbortSignal.timeout(5000)`**

Reasoning:

- Small Dutch SMB homepages (marfa.nl, similar Wix/WordPress sites) serve their homepage HTML without bot detection
- Scrapling's `SCRAPLING_BASE_URL` defaults to `http://localhost:3010` — not available in test environments, adds a dependency on the scrapling service being up
- The scrapling timeout is 30s — excessive for a homepage meta-tag scrape
- `lib/enrichment/favicon.ts` already uses plain `fetch` with `AbortSignal.timeout(3000)` for the same category of requests — same pattern applies
- If a homepage blocks plain fetch (rare for NL SMBs), og-logo returns null and falls through to favicon — graceful degradation

### Q3: What HTML parser is available?

**jsdom v27.4.0 is in `package.json` devDependencies** (confirmed at package.json line 88). However:

- It is a dev dependency — not available in production server code
- The vitest config sets `environment: 'jsdom'` globally — it's the test environment

**For server-side og-logo.ts: use regex-based extraction.** The og:image and similar tags are predictable enough that regex is reliable and avoids any parser import. A single regex with non-greedy matching on attribute content is sufficient. Example patterns:

```
/<meta[^>]+property=["']og:image["'][^>]+content=["']([^"']+)["']/i
```

Also handle reversed attribute order:

```
/<meta[^>]+content=["']([^"']+)["'][^>]+property=["']og:image["']/i
```

### Q4: What does /admin/prospects/new look like currently?

**Key findings:**

- File: `app/admin/prospects/new/page.tsx` — 259 lines, single `default function NewProspect()`
- Calls `api.admin.createAndProcess` (NOT `api.admin.createProspect`)

## Open Questions

1. **Apollo 422 early retry exit**
   - What we know: `apolloFetch` retries 3 times (MAX_RETRIES=3) on all errors except 429. A 422 no-coverage will be retried twice unnecessarily.
   - What's unclear: Is the 2s overhead acceptable, or should we fast-exit on 422?
   - Recommendation: Add `if (response.status === 422) break;` inside the apolloFetch retry loop to skip retries for no-coverage. This is a cheap improvement.

2. **og:image on Wix-hosted SMB sites**
   - What we know: Marfa uses Wix. Wix sites do set og:image (the site's own first page image).
   - What's unclear: Quality varies — Wix og:image may be a product/team photo rather than a logo.
   - Recommendation: Accept any og:image for now. If quality is poor after real usage, extend with a size-preference heuristic (prefer square images).

3. **createProspect vs createAndProcess for D3**
   - What we know: The form page calls `createAndProcess`. The "Optionele verrijking" fields belong on the form page.
   - Decision needed: Should BOTH mutations be extended, or only `createAndProcess`?
   - Recommendation: Extend both. `createAndProcess` for the form. `createProspect` for API callers. Same Zod fields, same sticky logic.
