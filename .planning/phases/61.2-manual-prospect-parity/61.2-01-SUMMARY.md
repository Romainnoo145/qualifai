# Phase 61.2 Plan 01: Apollo No-Coverage Error Handling Summary

**One-liner:** Typed `EnrichmentNoCoverageError` sentinel converts Apollo 422 "no org found" from a red crash into an amber partial-success that flows through to the Acties panel automatically.

## What Was Built

### Classes and Sentinels

**`EnrichmentNoCoverageError`** in `lib/enrichment/providers/apollo.ts`:

- Exported class extending `Error` with `name = 'EnrichmentNoCoverageError'` and `domain: string` property
- Mirrors `EnrichmentPlanLimitedError` pattern in service.ts
- Re-exported from `lib/enrichment/service.ts` so callers import from a single place

### Apollo Provider Changes (`lib/enrichment/providers/apollo.ts`)

- `apolloFetch`: 422 responses now fast-exit the retry loop (`noRetryError` flag bypasses catch-with-backoff) — saves ~2s per 422 attempt
- `enrichCompanyWithFallbackQueries`: catches 422 pattern in `lastError.message`, breaks the fallback query loop on first 422 (same domain with different params would also 422), then throws `EnrichmentNoCoverageError(clean)` instead of the raw Apollo error

### Service Layer Changes (`lib/enrichment/service.ts`)

- `runWithWaterfall` gains `noCoverageCount` counter alongside existing `errorCount`/`noResultCount`/`planLimitCount`
- `EnrichmentNoCoverageError` is caught distinctly in the provider loop — increments `noCoverageCount`, does NOT add to `errors[]`, does NOT increment `errorCount`
- After the loop: if `noCoverageCount > 0 && errorCount === 0 && noResultCount === 0 && planLimitCount === 0`, returns a minimal `EnrichedCompanyData` with just `domain` populated (all other fields null/empty arrays) instead of throwing
- `runWithWaterfall` accepts optional `input?: { domain: string }` parameter so the partial-success path can echo back the original domain

### Admin Router Changes (`server/routers/admin.ts`)

- Imports `EnrichmentNoCoverageError` from `@/lib/enrichment/service`
- `enrichProspect` wraps `enrichCompany` in try/catch for `EnrichmentNoCoverageError`
- On no-coverage: sets `noCoverage = true`, fills minimal `enriched` object, continues to `mergeApolloWithKvk` (KvK enrichment still runs), then returns `{ success: true, fallbackUsed: true, noCoverage: true }`
- On normal success: returns `{ success: true, fallbackUsed: false, noCoverage: false }`
- Previously returned the full prospect row; now returns the structured shape the Acties panel expects

### Return Shape on No-Coverage Path

```typescript
{ success: true, fallbackUsed: true, noCoverage: true }
```

The existing `markSuccess(data)` hook in `ProspectActionsPanel` reads `data.fallbackUsed`:

- `true` → `setState('fallback')` → amber state renders automatically
- No component change required

### Error Mapping (`components/features/prospects/error-mapping.ts`)

New constant:

```typescript
export const FRIENDLY_ERROR_APOLLO_NO_COVERAGE =
  'Apollo heeft deze organisatie niet in zijn database — verrijking gedeeltelijk.';
```

This is a positive-success warning copy for the amber state — NOT returned by `mapMutationError`.

## Deviations from Plan

### Auto-fixed: enrichProspect return shape change

**Found during:** Task 2
**Issue:** The original `enrichProspect` returned the full prospect row (`updated`). Changing to `{ success, fallbackUsed, noCoverage }` required updating both the normal-success path and the early-return "enrichment fresh" path.
**Fix:** Both return paths updated to return the structured shape. The "enrichment fresh" early return now returns `{ success: true, fallbackUsed: false, noCoverage: false }` for consistency.
**Files modified:** `server/routers/admin.ts`

### Note: EnrichedCompanyData has more fields than plan showed

The plan's interface snippet omitted `technologies`, `specialties`, `naicsCode`, `sicCode`, `lushaCompanyId`, `intentTopics`, `fundingInfo`, `rawData`. These are present in `lib/enrichment/types.ts`. All minimal-data objects (in both `service.ts` and `admin.ts`) include these fields as empty arrays / null values to satisfy the type. The `as unknown as T` cast is used in `runWithWaterfall` because the generic return type `T` is not constrained to `EnrichedCompanyData` — this is acceptable given the conditional exit is only triggered in the `enrichCompany` call path.
