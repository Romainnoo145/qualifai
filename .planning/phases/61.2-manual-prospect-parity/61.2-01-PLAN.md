<objective>
Prevent Apollo 422 errors from blocking the entire enrichment pipeline for small Dutch SMBs outside Apollo's database coverage. Instead of a red error toast that leaves Romano confused and the prospect in a broken state, introduce a typed `EnrichmentNoCoverageError` sentinel that travels up the call stack as a partial-success signal — the Acties panel amber branch (already built in Phase 61.1) fires automatically.

Purpose: Marfa smoke test revealed that "Verrijk opnieuw" renders a red crash toast for any prospect outside Apollo's DB. The real message is "Apollo doesn't know this company" — a recoverable, expected state for small Dutch SMBs.

Output:

- `EnrichmentNoCoverageError` class in `lib/enrichment/providers/apollo.ts` (mirrors `EnrichmentPlanLimitedError` pattern in same file)
- `enrichCompanyWithFallbackQueries` throws `EnrichmentNoCoverageError` instead of re-throwing the 422 error
- `runWithWaterfall` in `service.ts` catches `EnrichmentNoCoverageError` distinctly and returns a minimal `EnrichedCompanyData` with just `domain` populated
- `admin.enrichProspect` catches `EnrichmentNoCoverageError` and returns `{ fallbackUsed: true, noCoverage: true }` — the existing `markSuccess(data)` with `data.fallbackUsed` in `ProspectActionsPanel` fires the amber branch automatically
- `FRIENDLY_ERROR_APOLLO_NO_COVERAGE` constant added to `error-mapping.ts`
- Unit tests covering the 422 sentinel path and the unrelated-error throw path
  </objective>
