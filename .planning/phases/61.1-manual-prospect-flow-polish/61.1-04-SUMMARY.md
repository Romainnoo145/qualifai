- `ProspectActionsPanel` component with three independent mutation buttons: "Verrijk opnieuw" (enrichProspect), "Run research" (runResearchRun), "Run analyse" (runMasterAnalysis); each button cycles idle/loading/success/error with Dutch labels

## Parked Follow-ups — Phase 61.2 "Manual Prospect Parity"

The following 4 issues were discovered during smoke testing and formally parked into Phase 61.2. They are NOT blockers for Phase 61.1 completion.

1. **Marfa favicon quality (61.2-02)** — Marfa.nl only has a 16x16 Wix default .ico; upscaled it renders as a generic globe. Correct fix requires scraping og:image as a higher-quality fallback before falling through to the favicon chain.

2. **Apollo 422 on small Dutch SMBs (61.2-01)** — "Verrijk opnieuw" returns 422 for Marfa because Apollo doesn't cover small Dutch SMBs. Currently surfaces a generic error. Requires Apollo error swallowing + graceful "niet beschikbaar" state in the Acties panel.

3. **Manual prospect detail card is hollow (61.2-02 + 61.2-03)** — Fields like industry, description, location, employees, revenue, and LinkedIn are only populated via the Apollo path. Manual prospects have none of these. Requires an inline edit form + og:image/meta scraping for the non-Apollo path.

4. **Evidence/analysis pages need parity audit for missing-Apollo-fields rendering (61.2-04)** — Once the above fields can be populated for manual prospects, the existing evidence and analysis pages need a pass to ensure they render gracefully when Apollo fields are absent.

## Decisions Made

- Three retrigger mutations kept as separate `useMutation` hooks inside `ProspectActionsPanel` — each has distinct loading state, label, and error surface; abstraction would obscure the per-button state machine.
- `FRIENDLY_ERROR_GEMINI_FALLBACK` shown as amber toast in Acties panel on the success path (`fallbackUsed===true`), and as static indicator in `ProspectLastRunStatus` — two surfaces, same constant, no duplication of copy.
- DuckDuckGo ip3 promoted to primary favicon source after smoke test revealed Google HEAD probe failed for smaller Dutch domains while DuckDuckGo returned valid URLs (Rule 1 auto-fix).
- Marfa favicon quality issue formally parked to Phase 61.2 — fixing it correctly requires og:image scraping which is architectural in scope for Plan 04.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Flipped favicon probe chain — DuckDuckGo primary, Google fallback**

- **Found during:** Post-checkpoint smoke test (Romano's manual verification)
- **Issue:** Google s2/favicons HEAD probe was returning failures for several smaller Dutch domains that DuckDuckGo ip3 served correctly; the previous primary/fallback order was wrong for the actual prospect base
- **Fix:** Swapped probe order in `getFaviconUrl` — DuckDuckGo ip3 now probed first, Google s2 attempted only on DuckDuckGo failure
- **Files modified:** `lib/enrichment/favicon.ts`
- **Commit:** `0cdb4df`
