<current_state>
The evidence pipeline is Qualifai's core. Every prospect voorstel (discover page), offerte, and outreach is built on evidence. But the pipeline has structural problems that cause low-quality output.

Two root problems were identified in session 2026-04-20:

1. **De trechter is lek** — evidence items enter the DB without quality filtering. 404 pages, duplicate content, cookie banners, irrelevant "over ons" text, all get stored as EvidenceItem and passed to the AI.

2. **De masterprompt doet te veel** — one Gemini call generates: opening hook, executive summary, 3-5 narrative sections with punchlines, visualType + visualData per section. That's 4-5 tasks in one prompt.

These reinforce each other. Dirty input + overloaded prompt = inconsistent output.
</current_state>

## The Evidence Trechter (Funnel) — Current State

### How evidence enters the DB

```
Apollo enrichment → prospect profile (industry, employees, etc.)
                ↓
Source discovery → sitemap URLs + SERP URLs + default URLs (17 paths)
                ↓
URL selection → top N URLs ranked by relevance
                ↓
web-evidence-adapter.ts → fetch each URL → extract evidence drafts
                ↓
research-executor.ts → store as EvidenceItem rows
```

### Where it leaks

1. **404 pages (PARTIALLY fixed)**
   - Stealth path: now checks `statusCode >= 400` → skip. FIXED.
   - Crawl4AI path: has `looksLikeCrawled404()` check but NO HTTP status check. STILL LEAKS.
   - Fallback drafts: when stealth fails AND browser budget exhausted, creates a fallback EvidenceItem with just the URL and no content. STILL LEAKS.
   - `riskySeedGuess` flag exists but only skips fallbacks, not actual 404 content.

2. **Duplicate content**
   - Same page scraped via sitemap AND serp AND default paths → 3 EvidenceItems with identical snippets
   - URL dedup exists in `source-discovery.ts` but content dedup does NOT exist
   - A prospect can have 400+ evidence items where 60% are duplicates

3. **Irrelevant content**
   - Cookie banners, privacy policies, generic "over ons" text stored as evidence
   - No relevance scoring at ingestion time — everything with >500 chars gets stored
   - The `confidenceScore` in EvidenceItem is static per sourceType (0.70 for WEBSITE, 0.78 for REVIEWS), not content-quality based

4. **No pre-prompt evidence ranking**
   - The `.slice(0, 60)` in the masterprompt is arbitrary — it takes the top 60 by confidence, but confidence is sourceType-based not content-based
   - STB-kozijnen: 233 items, top 60 sent to AI, 173 ignored. No guarantee the 60 are the best 60.
   - Mujjo: 427 items. Same problem at larger scale.

### The Masterprompt — Current State (after today's changes)

```
SYSTEM_PREAMBLE (role definition)
TONE_RULES (style instructions)
PROSPECT PROFILE (from Apollo)
EVIDENCE (top 60 items × 300 chars = ~18k chars)
CROSS-CONNECTIONS (other prospects mentioned)
ADAPTIVE INSTRUCTIONS (conditional)
OUTPUT FORMAT (JSON schema for sections + visualType/visualData)
```

Still one prompt doing: narrative writing + visual data selection + citation threading + JSON structuring.

The visualType/visualData spec is ~30 lines of JSON schema that Gemini frequently ignores or gets wrong.

### The Legacy v1 Prompt (dead code)

`buildLegacyPrompt` (~260 lines) has NO callers. `analysis-v1` is deprecated. Safe to delete.

## Milestone scope — Evidence Pipeline Overhaul

This is the core of the app. Suggested approach: `/gsd:new-milestone` to properly scope this.

### Stream 1: Evidence Quality (the trechter)

**Goal:** Every EvidenceItem in the DB is a high-quality, deduplicated, relevant piece of information.

- Fix remaining 404 leaks (Crawl4AI path, fallback drafts)
- Content deduplication at ingestion (hash-based or semantic)
- Relevance scoring at ingestion — AI-score each item for "is this useful for workflow analysis?" (Gemini Flash, cheap)
- Drop items below threshold before storing
- Evidence summary per prospect: pre-compute a ranked, deduplicated set for the masterprompt

### Stream 2: Masterprompt Simplification

**Goal:** The masterprompt generates ONLY the narrative. Everything else is downstream.

- Delete legacy v1 prompt (260 lines of dead code)
- Remove visualType/visualData from masterprompt → separate Gemini Flash call per section
- Pre-filter evidence to top 20 highest-quality items, grouped by sourceType
- Simplify JSON output schema to just: openingHook, executiveSummary, sections[{id, title, body, citations, punchline}]

### Stream 3: Pipeline Automation Verification

**Goal:** From Apollo call → evidence → analysis → voorstel is fully automated and produces consistent quality.

- Run the full pipeline for 3-5 existing prospects with the new evidence filtering
- Compare output quality before/after
- Verify the discover page renders correctly with the simplified analysis output
